# PartyCRM Roadmap

Рабочее название продукта: `PartyCRM`.

Документ описывает план создания второго продукта на основе опыта и ядра ArtistCRM. Название, домен и финальное позиционирование могут быть изменены до запуска.

---

## Продуктовая гипотеза

PartyCRM — CRM для компаний, которые управляют заявками, площадками, сотрудниками, бронированиями, оплатами и подготовкой мероприятий.

Первичные сегменты:

- праздничные агентства;
- детские игровые и развлекательные точки;
- анимационные команды;
- event-команды;
- фотостудии и студии с несколькими залами/площадками;
- площадки под мероприятия и мастер-классы.

Главное отличие от ArtistCRM:

- ArtistCRM решает задачу соло-артиста: не потерять заявку, контакт, оплату и событие.
- PartyCRM решает задачу компании: контролировать заявки, точки, администраторов, исполнителей, бронирования и операционную подготовку.

---

## Стратегия продукта

- Бренд, домен, лендинг, тарифы и onboarding должны быть отдельными от ArtistCRM.
- На первом этапе не делать полный форк кодовой базы.
- Использовать текущий проект как технологическую платформу и постепенно выделять общий core.
- Данные PartyCRM и ArtistCRM хранить в разных БД/DB namespace с отдельными переменными подключения.
- Не усложнять ArtistCRM функциями для компаний.
- Новые company-функции проектировать так, чтобы они могли жить в отдельном интерфейсе и позже быть вынесены в отдельное приложение/репозиторий.

Целевое состояние:

```txt
ArtistCRM
- solo UI
- личные заявки
- личные мероприятия
- простые оплаты и документы

PartyCRM
- company UI
- точки/залы/площадки
- сотрудники и роли
- назначения исполнителей
- бронирования и конфликты
- выплаты исполнителям
- чеклисты администраторов

Общее ядро
- auth/tenants
- clients
- events/orders
- payments/transactions
- documents
- reminders
- public leads API
```

## Архитектура данных

Принятое решение: не смешивать данные ArtistCRM и PartyCRM в одной базе данных.

На старте:

```txt
ArtistCRM
- домен: artistcrm.ru
- БД: ArtistCRM production database
- env: MONGODB_URI / MONGODB_DBNAME

PartyCRM
- домен: partycrm.ru
- БД: PartyCRM production database
- env: PARTYCRM_MONGODB_URI / PARTYCRM_MONGODB_DBNAME
```

Правила:

- В коде должен быть явный product/database context: `artistcrm` или `partycrm`.
- Нельзя выбирать БД неявно только по пользовательским данным.
- Все API PartyCRM должны работать только с PartyCRM DB.
- Все API ArtistCRM должны продолжать работать только с ArtistCRM DB.
- Общие модели можно переиспользовать, но подключение к Mongoose/MongoDB должно быть product-aware.
- Даже при отдельной БД сохраняем `tenantId` во всех tenant-owned сущностях PartyCRM, потому что внутри PartyCRM будет несколько компаний.
- Миграции, seed-данные, backup и restore должны выполняться отдельно для каждой БД.

Причина решения:

- меньше риск случайного смешивания клиентских данных;
- проще юридически и операционно объяснять изоляцию продуктов;
- проще будет выделить PartyCRM в отдельный проект/репозиторий, если продукт подтвердится;
- меньше риск, что company-схемы и нагрузки начнут влиять на solo-продукт.

## Что не входит в MVP

- Полноценный склад и учет реквизита.
- Сложный график смен.
- Зарплатные ведомости с налогами и бухгалтерией.
- Франшизы и многоуровневые филиалы.
- Брендирование под клиента.
- Кастомные права доступа до уровня отдельных кнопок.
- Сложная BI-аналитика.

---

## Открытые вопросы

- Оставлять ли финальное название `PartyCRM` или выбрать более широкое B2B-название.
- Делать ли первый запуск под доменом `partycrm.ru` или временно под поддоменом существующего продукта.
- Нужен ли отдельный мобильный кабинет исполнителя в первом релизе или достаточно responsive web/PWA.
- Нужна ли поддержка нескольких залов внутри одной точки в MVP.
- Нужно ли разрешить бронирование без клиента: техническое закрытие времени точки.
- Какие клиентские данные исполнитель может видеть по умолчанию.
- Нужен ли статус подтверждения исполнителем назначения в MVP.
